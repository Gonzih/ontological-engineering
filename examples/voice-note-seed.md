# Example: Voice Note As Seed Knowledge

This example shows how the initial direction-setting voice note becomes structured knowledge. The goal is not to capture every word. The goal is to preserve the conceptual movement from raw intention to ontology, epistemology, research agenda, and repository structure.

## Raw Input

Source: founder voice note, July 8, 2026.

Summary: The repository should move beyond article-style research about ontological engineering and become a deeper inquiry into company knowledge representation, epistemology, agentic knowledge ecosystems, and specification-driven software generation. The system should model a company's second brain as time-aware, hierarchy-aware, relationship-aware, and capable of preserving the evolution of thinking.

Status: `raw`

## Extracted Direction

The note reframes the project.

Earlier framing: write pages and articles about ontological engineering.

New framing: develop ontological engineering as a domain for designing the knowledge substrate of an AI-native company.

This creates a shift from content production to infrastructure thinking. The repository should still contain writing, but the writing should clarify a system that can later become schemas, workflows, agent roles, and software specifications.

## Extracted Concepts

### Ontological Engineering

Working definition: the practice of structuring a company's second brain so knowledge is time-aware, hierarchy-aware, relationship-aware, and usable by humans and AI agents.

Relationships:

- depends on ontology
- depends on epistemology
- enables specification-driven software
- governs agentic knowledge ecosystems

### Company Second Brain

Working definition: the evolving knowledge representation of the company, including concepts, claims, evidence, decisions, rationale, specifications, agent outputs, artifacts, and history.

Important distinction: this is not a document archive. It is structured organizational memory.

### Agentic Knowledge Ecosystem

Working definition: the system of human and AI actors that generate, acquire, process, prioritize, validate, and apply knowledge.

Key risk: agent-generated information can become organizational knowledge accidentally unless promotion status and provenance are explicit.

### Specification-Driven Software

Working definition: designing software by first specifying user interactions, agent interactions, constraints, evidence, assumptions, and conceptual models, then generating one or more software iterations from that specification.

Key implication: the quality of generated software depends on the quality of the specification and the knowledge system behind it.

## Extracted Claims

### Claim 1

Proposition: Ontological engineering for a company is less about writing pages about ontology and more about designing the knowledge substrate of the organization.

Status: `accepted`

Scope: repository direction.

### Claim 2

Proposition: The company knowledge base must preserve the evolution of thinking, not only the latest conclusions.

Status: `accepted`

Scope: core design principle.

### Claim 3

Proposition: Software should be generated from specifications that capture user interaction, agent interaction, and conceptual intent.

Status: `proposed`

Scope: product architecture thesis.

Evidence basis: founder direction and current AI software-generation context.

### Claim 4

Proposition: AI agents require explicit epistemic boundaries because they generate, acquire, process, and prioritize information.

Status: `accepted`

Scope: agentic knowledge ecosystem design.

## Extracted Decisions

### Decision 1

Decision: Organize the repository around foundational philosophy, company memory, agentic systems, specification-driven software, schemas, and examples.

Rationale: The domain spans ontology, epistemology, knowledge engineering, product specification, and AI agent governance. Separate but connected documents allow the system to grow coherently.

### Decision 2

Decision: Treat time, hierarchy, and relationship awareness as first-class requirements.

Rationale: The founder note emphasizes evolution of thinking and relationships between pieces of knowledge. These cannot be added as metadata later; they shape the whole system.

## Open Questions

- What schema is expressive enough for company wisdom without becoming too heavy to use?
- Which entities should be canonical in the first implementation?
- How should agent-generated outputs be prevented from polluting accepted knowledge?
- How should time-aware knowledge be queried?
- How should specifications be evaluated before software generation?
- How should disagreement and unresolved ambiguity be represented?
- How should software artifacts link back to the decisions and claims that produced them?
